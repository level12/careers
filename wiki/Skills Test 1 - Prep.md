## App Template

You may prepare, ahead of the test, a base project/template that covers the fiddly setup
work but does not pre-build the product.

Do:

- Use code generation utilities to lay down the app basics.
  - Examples: a Django project generator, `create-react-app`, Vite, etc.
- Set up standard dev tooling like `mise`, `uv`, linting/formatting, test config, and
  similar project plumbing.
- Set up the basic frontend/backend structure and connection points.
- Keep the template generic and reusable.

Don't:

- Anticipate product features and build them ahead of time.
- Add libraries, integrations, or setup for anticipated project needs that would not
  belong in a generic base template.
- Include product-specific models, workflows, UI, business logic, or other domain code.
- Turn the template into a partial solution.

In short: it should remain a skeleton that removes setup friction, not a head start on the
actual assignment.

## Commits / Branches

- If you are going to use any kind of boilerplate app setup/template, commit that
  **directly to the main branch**. This is usually a lot of files and should be
  mostly/exclusively code generated by other tools, so we don't want that noise in the
  main PR.
- The features/functionality you add should be in a feature branch and PR for easier
  review.

## Technologies

- Backend: Python, use whatever libs/framework you want
- Frontend:
  - Use whatever you are most comfortable and productive with for the test
  - Site should feel/be modern and helpfully dynamic but does not need to be a SPA
  - Slightly prefer solutions that don't need a front-end build step due to dependency
    churn and supply chain risk. But, your productivity is more important than this
    consideration so, again, **choose what you will be most productive in.**
- Dev tools: mise, uv
  - That is, another developer with these tools installed should be able to fully run and
    test the application without having to manually install or have present on their dev
    machine any other apps, libraries, or tooling.
  - This also facilitates the CI setup being relatively simple.
- Services: database, etc.
  - Prefer those to be defined in a Docker Compose file.
- Third party services: should generally avoid

## Time Limits & Time Keeping

Test time limit: 4 hours

Time spent reading/understanding/discussing the instructions **does not count towards your
time limit**.

- The time limit is 4 hours. Not 4:15 or even 4:05.
- Make sure you have a commit of your final work within the time limit given.
- You may take additional time to submit the PR and update `feedback.md` (see below).

We expect you to start taking the tests promptly upon receiving them and then work them to
completion with only a small break midway through if needed.

We realize this aspect of the testing process is a departure from real-world development.
But we need some consistency in the testing process in order for our evaluations to be
uniform across candidates.

## When Finished (Feedback)

After you have made your final commit on the app inside the time limit, create a
`feedback.md` file in the repo with any feedback you have.

Please reflect on anything that you think would be helpful in our evaluation and/or any
ideas you might have on improvements we could make to the test or process.

This time does not count against the time limit given above as it should occur once said
limit has expired.
